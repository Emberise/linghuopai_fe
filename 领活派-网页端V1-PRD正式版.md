## 1. 文档目标

本文档用于将领活派从“Demo 可演示状态”推进为“可正式研发落地的 V1 版本定义”，明确以下内容：

1. 领活派要解决的核心问题与 Why Now；
2. V1 正式开发阶段的产品定位、范围边界与成功标准；
3. 普通用户端、企业端、平台管理后台三端的页面结构与主业务流程；
4. AI / Agent 在产品中的职责边界、输出形态与状态流；
5. 与研发实现直接相关的功能要求、非功能要求、优先级与风险项。

本文档的目标不是继续描述一个“活动 Demo”，而是形成一份可用于正式研发、评审、设计和测试协同的正式 PRD。

## 2. 相较整合版 PRD 的关键升级与取舍

本版在保留《领活派-网页端V1-PRD整合版》业务方向的基础上，吸收 StellaWork 正式版 PRD 的写法、页面结构与业务表达方式，形成以下关键升级：

### 2.1 从 Demo 导向，升级为正式开发导向

整合版 PRD 更偏向“首期活动 + Demo 交付”语境；本版转为正式开发语境，核心关注：
- 页面体系是否完整；
- 状态与权限是否可建模；
- AI 机制是否可落到数据对象与流程上；
- 管理后台是否具备最小运营能力。

### 2.2 从“功能罗列”升级为“页面 + 流程 + 状态”三层定义

参考 StellaWork 的正式版结构，本版将需求拆为：
- 页面与信息架构；
- 主流程；
- 模块需求明细；
- 状态模型；
- 业务规则；
- 非功能要求。

### 2.3 明确纳入平台管理后台入口与最小后台范围

参考 StellaWork 登录页与后台结构，本版明确：
- 登录页需要提供“平台管理后台入口”；
- V1 需要最小化管理后台，而不是仅保留概念；
- 管理后台至少覆盖：系统监控与看板、零工用户管理、企业客户管理、画像数据管理、匹配结果与调度。

### 2.4 保留统一账号思路，但强化角色权限

参考 StellaWork 当前“同一工作台兼具人才侧与企业侧入口”的设计，同时结合领活派现有需求，本版确定：
- 账号底座统一；
- 页面能力按角色控制；
- 首页可保留“普通用户主视角 + 企业快捷入口”的工作台模式；
- 企业、普通用户、平台管理员在数据权限上必须严格隔离。

### 2.5 引入更完整的企业侧闭环表达，但保留范围边界

StellaWork 已体现出“发布需求 → AI 结构化 JD → 推荐候选人 → AI 初筛报告 → 直聊/发放任务”的强链路表达。

领活派 V1 正式版吸收这套表达方式，但做如下取舍：
- **纳入 V1 正式范围**：发布需求、AI 结构化、候选人推荐、AI 初筛报告、站内沟通；
- **作为条件能力/P2 预留**：任务发放、托管金额、签约、B2C 结算、Excel 回流、轻量验收。

### 2.6 明确领活派与 StellaWork 的业务差异

虽然可大量参考 StellaWork 的 UI 与业务逻辑，但两者不完全等同：
- StellaWork 偏技术人才 / 技术项目撮合；
- 领活派 V1 面向更广泛的灵活用工与项目协作场景，优先覆盖设计、内容/运营支持、前端/技术协作、活动执行与短期支持。

因此，领活派正式版在结构上参考 StellaWork，在业务范围上保持自身定位。

## 3. 项目背景与 Why Now

### 3.1 项目背景

灵活用工、兼职、零工、短期协作与项目制合作场景中，企业、劳动者与平台三方都存在明确效率问题。

#### 企业侧问题
- 原始 JD / 任务说明不完整，表达方式不统一；
- 需求往往以聊天记录、口头描述、文档片段存在，结构化成本高；
- 企业前期需要重复解释需求、反复筛人，沟通成本高；
- 传统平台偏信息分发，缺少对“需求本身”的理解与标准化。

#### 普通用户侧问题
- 用户有能力，但不会标准化表达；
- 简历、作品、证书、可服务时间等资料分散；
- 用户不清楚自己适合哪些任务，也不清楚企业真正重视什么；
- 报名后缺少明确反馈与推进状态。

#### 平台侧机会
如果平台能够建立以下三类能力，就具备明显差异化空间：
- **需求理解能力**：把模糊需求转成结构化任务；
- **能力表达能力**：把分散资料转成画像与标签；
- **前置筛选能力**：用 AI / Agent 替代一部分重复沟通。

### 3.2 Why Now

领活派当前进入 V1 正式开发阶段，时机已经成熟，原因包括：

1. **双边撮合 Demo 已完成验证**：核心链路已跑通，当前需要产品化、权限化、状态化，而不是继续做演示拼装；
2. **企业侧与劳动者侧流程都已相对清晰**：现在更需要把页面、对象、状态和后台能力沉淀为正式系统；
3. **AI 非常适合承接前置高重复环节**：需求补全、资料采集、标准化问答、结构化总结已经具备可落地条件；
4. **正式开发要求后台与运营能力补齐**：从 Demo 进入正式产品，必须纳入管理后台、数据监控、权限控制与状态闭环；
5. **需要一份统一正式版 PRD 支撑研发协同**：避免后续继续在零散说明、补充文档、临时聊天中推进。

## 4. 产品定位

### 4.1 产品定义

**领活派网页端 V1 是一个 AI 驱动的双边撮合平台，面向灵活用工、兼职、零工与项目制协作场景，帮助企业更高效发布任务、筛选候选人，也帮助普通用户更低成本表达能力、报名任务并进入标准化沟通流程。**

### 4.2 核心价值主张

#### 面向企业
- 降低任务 / JD 录入与表达成本；
- 自动完成需求补全与结构化；
- 快速获得 AI 推荐候选人与初筛结果；
- 减少前期重复沟通与无效筛人。

#### 面向普通用户
- 低门槛补充简历与能力信息；
- 获得与自身更匹配的任务推荐；
- 通过 AI 辅助与 AI 初筛更快进入企业沟通；
- 在“我的任务”中看到明确推进状态。

#### 面向平台运营
- 将“需求理解—能力表达—AI 初筛—企业推进”沉淀为标准化流程；
- 建立匹配数据、会话数据、转化数据与画像数据资产；
- 形成可持续优化的人岗匹配基础设施。

### 4.3 V1 的核心切入方式

V1 不追求完整灵活用工交易闭环，而是聚焦把以下链路做深做通：

> 企业发布任务 → AI 补全需求 → 用户补充资料 → 用户报名 / 投递 → 企业 Agent 完成标准化 AI 初筛 → 企业查看结构化结论 → 平台沉淀匹配与转化数据

## 5. 产品目标与成功标准

### 5.1 V1 核心目标

V1 需要验证并正式落地以下 6 件事：

1. 企业是否愿意在平台持续发布任务 / JD；
2. 普通用户是否愿意在平台完成资料补充并参与报名 / AI 面试；
3. 企业 Agent 是否能显著降低企业前期沟通成本；
4. 平台 AI 是否能提升用户资料完整度与推荐体验；
5. 平台能否让企业更快进入“值得真人沟通的人”；
6. 平台管理后台是否能支撑基础运营、管理、监控与匹配调度。

### 5.2 北极星指标

- **已发布任务在 72 小时内进入“企业查看 AI 初筛结果 / 发起真人沟通”阶段的占比**

### 5.3 企业侧指标

- 单任务从原始描述到结构化任务的平均耗时；
- 企业完成发布需求的成功率；
- AI 推荐候选人被企业查看的比例；
- 企业查看 AI 初筛报告率；
- 企业从发布任务到首次真人沟通的平均耗时；
- 企业主观认可的匹配结果占比。

### 5.4 用户侧指标

- 从进入平台到完成报名 / 投递的转化率；
- 资料补全率；
- 简历上传率 / 作品集上传率 / 证书上传率；
- AI 初筛参与率与完成率；
- 推荐任务点击率；
- 进入企业真人沟通的比例。

### 5.5 平台侧指标

- 单任务有效候选池形成效率；
- 双向匹配率；
- 后台可统计的转化数据完整度；
- 管理后台使用率与关键数据可视化覆盖率；
- AI 输出可回溯率、状态流转可追踪率。

## 6. 目标用户与角色定义

| 角色 | 角色说明 | 核心诉求 |
| --- | --- | --- |
| 普通用户 / 零工人才 | 具备一定技能，希望获得兼职、零工或项目协作机会的人群 | 更快被理解、被推荐、被推进 |
| 企业需求方 | 有短期、兼职、灵活用工或项目协作需求的企业/团队 | 更快发任务、更快筛人、更少重复沟通 |
| 平台管理员 | 负责平台运营、管理、监控、数据维护与调度的内部角色 | 监控平台运行、管理数据、推动撮合 |
| 平台 AI | 面向普通用户的 AI 助手 | 推荐任务、补齐资料、答疑 |
| 企业 Agent | 面向企业的 AI 代理 | 补全需求、结构化 JD、标准化初筛 |
| 企业真人 | 企业真实业务人员 | 在 AI 初筛后做确认、沟通与推进 |

### 6.1 优先服务对象

#### 企业侧
- 中小企业与项目型团队；
- 需要设计、运营、前端、文案、活动执行等灵活人力的团队；
- 对“快速初筛候选人、减少重复沟通”有明确需求的企业。

#### 普通用户侧
- 自由职业者、兼职人群、项目协作人群；
- 设计、内容、运营、前端、活动支持等方向的劳动者；
- 愿意通过网页端快速报名并接受 AI 辅助的人群。

### 6.2 优先场景

V1 优先覆盖：
- 设计类任务；
- 内容 / 运营支持类任务；
- 前端 / 技术协作类任务；
- 活动执行与短期支持类任务。

### 6.3 暂不优先场景

- 高度依赖复杂排班与考勤的重运营场景；
- 依赖支付、税务、社保、金融服务的场景；
- 强依赖复杂验收、里程碑结算和系统深度对接的大客户项目。

## 7. 产品原则

1. **先验证并做深撮合链路，再扩展履约链路**；
2. **统一账号底座，角色权限严格隔离**；
3. **报名制优先，不做抢单制**；
4. **AI 负责前置提效，不替代最终业务判断**；
5. **资料采集以渐进式补全为主，不要求一次填完**；
6. **普通用户端重点适配手机 / Pad，企业端与后台重点适配 PC**；
7. **优先建设能被正式运营消费的数据与后台能力，而不是只做演示 UI**；
8. **为签约、发放、托管、验收预留接口和状态，不让其阻塞 V1 主链路。**

## 10. 核心业务流程总览

### 10.1 普通用户主流程

1. 用户扫码或访问网页进入登录页；
2. 完成登录；
3. 进入工作台首页；
4. 查看推荐任务或进入任务大厅；
5. 查看任务详情与 AI 匹配说明；
6. 若资料不完整，则补充简历 / 作品 / 时间 / 标签；
7. 用户报名 / 投递；
8. 企业 Agent 发起 AI 初筛；
9. 用户完成标准化问答；
10. 企业查看结构化结果；
11. 用户在“我的任务”中查看推进状态，并等待企业留言或真人沟通。

### 10.2 资料补全与画像生成流程

1. 用户进入“我的资料 / AI 画像”页；
2. 上传简历或通过对话补充经历；
3. AI 提取技能、项目经验与标签；
4. 生成能力画像；
5. 画像反哺任务推荐、匹配排序与初筛问题。

### 10.3 企业发布需求流程

1. 企业登录进入工作台 / 需求情况页；
2. 点击“发布任务”；
3. 上传任务文档 / PRD，或直接填写自然语言需求；
4. 企业 Agent 发起补问并自动结构化；
5. 企业确认结构化 JD；
6. 系统进入“等待 AI 分析 / 发现匹配人选”等任务状态；
7. 企业查看推荐候选人。

### 10.4 企业筛选与推进流程

1. 企业进入需求详情；
2. 查看 AI 匹配候选人列表；
3. 点击候选人详情；
4. 查看 AI 面谈总结、技能匹配、优势、风险和对话摘录；
5. 企业决定是否站内沟通 / 邀约 / 推进；
6. 若后续接口就绪，可进入发放任务 / 签约 / 托管等链路。

### 10.5 管理后台流程

1. 管理员从登录页入口进入后台；
2. 查看系统监控与平台总览；
3. 管理零工用户、企业客户与画像数据；
4. 查看匹配结果与调度状态；
5. 进行基础搜索、数据修正与运营动作。

## 12. AI / Agent 机制定义

### 12.1 四类核心交互关系

V1 明确存在以下四类会话 / 智能交互：

1. **普通用户 ↔ 平台 AI**：任务推荐、资料建议、问答辅助；
2. **企业 ↔ 企业 Agent**：需求补全、结构化任务生成、Agent 配置；
3. **普通用户 ↔ 企业 Agent**：标准化 AI 面试 / 初筛；
4. **普通用户 ↔ 企业真人**：合作细节确认与进一步沟通。

### 12.2 AI 角色定义

#### 平台 AI
面向普通用户：
- 推荐任务；
- 分析资料缺口；
- 提供报名建议；
- 回答求职 / 接单问题。

#### 企业 Agent
面向企业：
- 补全需求；
- 输出结构化任务；
- 根据企业偏好组织第一轮问题；
- 输出候选人结构化初筛结果。

#### AI 初筛面试官
面向用户：
- 基于岗位发起问题；
- 完成标准化问答；
- 为企业生成初筛报告。

#### AI 决策支持 Agent
面向企业：
- 输出建议、优势、风险；
- 给出技能维度分解；
- 给出对话摘录与证据。

### 12.3 AI 输出形态要求

所有关键 AI 输出都应结构化，至少包括：
- `summary`
- `tags`
- `riskFlags`
- `recommendation`
- `nextAction`

### 12.4 边界约束

- AI 不直接替企业做最终录用决策；
- AI 调用统一走服务端；
- Prompt 管理与工作流编排统一在后端完成；
- AI 输出必须可追踪、可回溯、可关联到任务与候选人对象。

## 14. 业务规则

### 14.1 任务来源规则
- 第一版任务统一由企业侧发布；
- 不对接第三方任务源；
- 不支持普通用户发布任务。

### 14.2 报名规则

- 企业有筛选权；
- 报名不等于接单成功。

### 14.3 资料补全规则
- 不要求首次填完全部资料；
- 在报名前、推荐前、面试前等节点触发补全提醒；
- 上传简历与对话式补充并行存在。

### 14.4 文件规则
- 简历、作品集、证书等文件需受权限控制；
- 文件必须与用户、候选人、任务对象绑定；
- 不允许匿名公开暴露。

### 14.5 AI 初筛规则
- 默认在用户报名后触发；
- 控制在 5 轮以内；
- 结果只用于辅助，不替代企业最终判断。

### 14.6 通知规则
- 用户报名、进入 AI 初筛、收到留言、被推进 / 未入选时应收到通知；
- 具体通知方式可先以站内消息为主，并预留短信 / 模板消息能力。

### 14.7 后台管理规则
- 管理后台必须保留操作日志；
- 后台管理动作应可回溯；
- 搜索、修改、调度等动作需有权限约束。

## 15. 非功能需求

### 15.1 性能与规模要求

第一版需满足以下基本规模目标：
- 用人单位规模：约 500 家，通过 PC 端录入 JD；
- 劳动者规模：1000+，通过手机扫码或网页进入；
- 峰值并发：1000 人同时使用；
- 关键页面首屏体验应稳定可用。

### 15.2 安全与权限

- 模型调用只发生在服务端；
- 模型 Key 不暴露给前端；
- 用户资料、文件、聊天记录、初筛结果受权限控制；
- 聊天与面试会话必须绑定任务、报名与角色关系；
- 企业、普通用户、管理员数据隔离必须明确。

### 15.3 可解释性

所有关键 AI 结果都应具备解释性，包括：
- 匹配度来源；
- 推荐理由；
- 技能匹配项；
- 风险提示；
- 执行建议。

### 15.4 可维护性

- AI 能力集中封装；
- 任务、报名、候选人、会话、文件等核心对象统一建模；
- 状态流统一维护；
- 后台操作与 AI 调用结果可追踪。

### 15.5 适配要求

- 普通用户端重点适配手机和 Pad；
- 企业端与管理后台重点适配 PC；
- 页面需支持扫码场景下的快速进入与首屏可用。

## 19. 风险与待确认项

### 19.1 产品风险
1. 若继续沿用 Demo 思维，正式开发时会出现权限、状态、后台缺失问题；
2. 若 AI 评估结论不稳定，会直接影响企业信任；
3. 若文档解析精度不高，会影响下游匹配质量；
4. 若后台只做占位而不做实质能力，正式运营会很快失真。

### 19.2 需求取舍待确认
1. 我的协议是否在 V1 实做，还是保留状态与入口；
2. 发放任务 / 托管金额是否纳入 V1 条件能力；
3. Excel 导入与结果回流是否进入本期；
4. 公共导航与品牌页是正式交付，还是仅作轻量承接页。

### 19.3 技术待确认
1. 多 AI 并发场景下的优先级、资源占用与调度策略；
2. 人岗匹配权重模型如何配置，企业是否支持任务级差异权重；
3. AI 对话是否支持用户打断；
4. 服务器资源与整体架构规模。

## 20. 结论

领活派网页端 V1 的正确方向，不是继续在 Demo 逻辑上堆页面，而是正式把以下三件事产品化：

1. **企业需求结构化**：让企业更快发任务、更快筛人；
2. **用户能力结构化**：让普通用户更容易被理解、被推荐、被推进；
3. **平台撮合流程结构化**：让 AI 补问、AI 初筛、状态流转与管理后台成为正式系统的一部分。

本版 PRD 参考了 StellaWork 的正式文档结构和线上 UI / 业务逻辑表达，吸收了其登录页、工作台、个人画像、需求管理、发布任务、候选人评估和平台管理后台等成熟模式；同时保留了领活派自身在灵活用工、多场景任务类型、统一账号、活动扫码与轻量履约预留方面的业务特点。

V1 的正式成功标准不是功能是否“大而全”，而是以下链路能否真实跑通并可持续运营：

> 企业发布任务 → AI 补全需求 → 用户补充资料并报名 → 企业 Agent 完成初筛 → 企业查看结构化结论并推进 → 平台后台沉淀和管理匹配数据

只要这条链路可以被真实业务消费、被后台管理、被数据追踪，领活派网页端 V1 就具备正式开发与持续迭代基础。
